# AerisHealthAssessment

## Folder Structure

### A Folder

- **main.py**: Solution for Question A
  - **Run**: Execute `python main.py` inside the A folder
  - **Requirements**: Python 3.x
- **QuestionAInputFile.txt**: Input file for test data
  - First line: Number of test cases (1-100)
  - Following lines: Each lines contains two integers x and n (1 ≤ x, n ≤ 10).

### B Folder

- **main.py**: Solution for Question B
  - **Run**: Execute `python main.py` inside the B folder
  - **Requirements**: Python 3.x
  - **Note**: Data validation logic is omitted for performance reasons. Input is assumed to be valid per problem constraints.
- **QuestionBInputFile.txt**: Input file for test data
  - First line: The first line contains an integer t (1 ≤ t ≤ 1000) — the number of test cases.
  - Following lines: Each of the lines contains one integer n (1 ≤ n ≤ 10¹⁸) — the total number of propulsion units.

### C Folder

- **APIQuestions.md**: Solution for frontend design
  - Refer to the file for details on Xero API implementation

### Frontend

- **Frontend Implementation Notes**:
  - While the task focuses on frontend technology, backend simulation is also implemented for versatility across multiple roles
  - Uses `localStorage` to simulate a database with interaction logic
  - Frontend requests data through simulated backend interfaces, not directly from `localStorage`
  - Implements lazy loading (verifiable via network requests in browser DevTools)
  - Router lazy loading was considered but not implemented due to time constraints and project scope
  - User authentication with token-based auto-login is implemented
  - Token encryption details use Asymmetric Encryption (available for further discussion in interviews)
- **Code Quality**:
  - Strict type constraints enforced throughout the project
  - Run `npm run lint:eslint` to verify code standards
  - High coding standards are maintained as an integral part of development
  - Unit tests were primarily generated by Claude Code; frontend/backend implementations were personally developed and tested
  - All unit tests pass validation
- **mock.ts**: Contains additional comments explaining implementation details, anticipated but unimplemented features, and unused technologies
